1.    Overview

RESQML is an industry initiative to provide open, non-proprietary, standard interfaces between software tools used to support geologic modeling and reservoir modeling.

1.1    Document Aims

This version represents an incremental set of fixes to version 1.0 and is intended to be a production version. A detailed list of schema changes since v1.0 can be found in the Change Detail file. This document provides a non-normative overview of the data schemas contained in this version. This document will list the data objects and will provide some insight to the file structure used in constructure used in constructing the schemas.

If there is any discrepancy between this document and the XSD files that use XML Schema syntax then the XSD files will prevail.

1.2    General Assumptions

It is assumed that the reader is:

  • Familiar with the syntax of the W3C's XML Schema Specification.
  • Familiar with the relevant Upstream Domain associated with each data schema.

While the above is not necessary for reading this document, it is necessary for understanding each of the data schemas.

1.3    Content Organization

This document is organized as follows:

  • Section 1: "Overview" is an introductory section setting the scope and the aims for this document. It also describes the target audience as well some key terms used in the document and how the document relates to the RESQML documentation set.
  • Section 2 "Data Schema Summary" provides a business overview of each data schema in the specication.
  • Section 3 "Schema Construction" provides an overview of the design methodology.
  • Section 4 "Data Schema Detail" provides a browsable view of the XSD files for each data schema.

1.4    Key Terms used in this Document

  • An XML Schema is a way to define the structure, content and, to some extent, the semantics of XML documents. RESQML utilizes the W3C's XML Schema Specification for defining schemas.
  • A Data Schema is a specific XML Schema which describes an XML document who's content represents a coherent set of data that is utilized in an exchange of data. The RESQML data schemas are constructed according to the WITSML schema construction guiddelines.
  • A Data Object is a term for XML data that conforms to the singular element that is a recurring child of a plural root element in a WITSML schema. It does not represent a formal programming object (with methods, properties, events, etc). A Data Object generally represents a coherent set of data that all represents details of the same concept. These tend to represent data that might otherwise be transmitted in a report or a spreadsheet and they commonly represent some sort of activity based business scenario. For RESQML the data object is called resqmlDocument.

2.    Industry Standards for Subsurface Model Exchange

Challenge: Integrating Technology to Enable Better, Faster Decisions

Reservoir studies have become more sophisticated, complex and interconnected—both upstream with seismic and geological data, and downstream with flow simulation benchmarked by fast time-lapse production data. These studies now also involve many more scenarios to help constrain uncertainties.

Typically performed with a variety of software applications on ever-shorter timelines for more challenging development programs, study results have improved dramatically due, in part, to continous improvement in computing performance. However, fast and efficient data exchange is also crucial for success.

In 1995, oil and gas companies and software vendors initiated the RESCUE Consortium to provide a multi-vendor E&P industry data exchange format for 3D gridded reservoir models, horizons, faults, structural models, and associated well data. Today we are extending RESCUE to provide an open earth model data exchange format using the XML technology, RESQML™.

What is RESQML?

RESQML is the data exchange format for transferring earth model data between applications in a vendor-neutral, open and simple format.

Benefits of RESQML

  • Fast and easy exchange
    Self-contained RESQML objects support flexible workflows for browsing or exchanging horizons, faults or grids. Today, with a combination of XML and Hierarchical Data Format (HDF) binary files, the structure and numerical details of the RESQML document can be read and understood by people and enables fast access and transfer of large, complex datasets.
  • Easy update
    The modular approach to RESQML design and development makes it easier to ensure data coherency and to focus on any step of model construction or update. For example, if needed, you can transmit only a fault—not the entire model.
  • Traceability
    The rich XML descriptions provide built-in documentation of objects and their uses. Metadata in XML is searchable and incorporates Dublin Core elements, which are cataloguing standards. It also leverages common data elements from other Energistics standards such as WITSML and PRODML.
  • Improved Grid Capabilities
    RESQML extends RESCUE to support unlimited numbers of multi-million cells in a model and unstructured grids. Grids are maintained in open binary formats for fast direct access. Fault representation is more explicit, eliminating geometric searches and ambiguities among vendors. More efficient sequencing means faster loading.
  • Geo-Referencing
    As in WITSML, all objects are positioned with respect to a Coordinate Reference System (CRS). Geography Markup Language (GML) usage ensures easy transfers to GIS aware databases. Objects may contain a GML bounding box to enable spatial indexing and searching.
  • Integration with WITSML and PRODML
    RESQML leverages WITSML and PRODML object descriptions, providing integrated access to well data and production data. Version 1 includes integration with these WITSML objects: well, wellbore, trajectory, formation marker, and well log. This work provides the foundation for integrating PRODML objects in later versions.

The Earth Model Framework

A goal is for the RESQML model to provide a structural framework comprised of horizons and faults, unconformity surfaces, and the chronological relationships between them.

Version 1 includes horizons and faults as individual structural features. Several geometrical representations of each will be supported including pointset, regular 2D grid, triangulated surface, and curve sets.

Transfer of horizons and faults does not require reference in a 3D grid, as did the RESCUE format.

The Cellular Reservoir Model

The RESQML model may also contain multiple cellular grids associated with the Earth Model and with the well data, which includes both static and dynamic data.

It provides shared data standards between static geocellular and dynamic simulation models.

Future Enhancements

RESQML benefits not only from the immense experience gained over the years with RESCUE, but also from the vision of new development.

RESQML will enable feedback loops from simulation to seismic, improving today’s usage of time-lapse seismic.

Gridding technologies such as unstructured gridding or micro-fracture descriptions, along with features like streamlines, are further enhancements that can be more easily integrated into RESQML.

The Energy Company Perspective

The new RESQML standard will enable development more easily than the RESCUE model. The flexibility of the XML code makes it possible to extend capabilities, and to more easily integrate applications (either in-house developed or purchased) for increased value along the entire E&P value chain.

The Software Developer Perspective

The modular aspect of RESQML will make developing and updating code easier and faster. Using standard exchange format - namely XML - means standard tools can be used thereby improving productivity and quality.

The Service Company perspective

With visualization being a vital component of today’s operations environment, an efficient standard format will provide easier access to subsurface data on the rig site. This, in turn, will enable representation of the reservoir and overburden where and when the well is drilled.

Special Interest Group (SIG)

The RESQML SIG is open to all Energistics members. Its function is to guide the evolution of RESQML, both by leveraging the knowledge gained from RESCUE, WITSML, and PRODML, and through the development and prioritization of enhancements worked from a use-case perspective.

The SIG is composed of a Steering Committee and two delivery teams, the Use Case Team and the Technical Team. Contact Us Today

If you are interested in joining the RESQML SIG or would like more information, please contact Jana Schey at Energistics (http://www.energistics.org/contact-us, Tel. +1 281.243.2133).

3.    Schema Construction

3.1    WITSML Introduction

The Wellsite Information Transfer Standard Markup Language (WITSML) is a standard for sending well site information in an XML document format between business partners. These RESQML schemas are crafted to be conformant to the design characteristics of WITSML version 1.4.0. XML schemas are used to define the content of an XML document. This section provides an overview of the Data Schema design. The RESQML specification consists of data object schema.It is anticipated that WITSML data objects will be utilized as needed. Specifically, the well, wellbore, trajectory, log and formationMarker data objects are expected to be use. A data object schema defines a set of data that can be transmitted within a single XML document. Data object schemas contain attributes, elements, and included component sub-schemas.

Component schemas may be in separate XML schema files, but do not represent complete data objects (becasue they do not contain global elements). A component schema may be included by more than one data object schema. All component schemas are prefixed with (cs_). Each component schema file generally defines one type that has the same name as the file name. While the normative schema utilizes multiple files, the RESQML Web Services will utilize derived schemas which are packaged with all components of a schema in one file.

An attribute group file (attgrp_uid.xsd) is used to define a unique identifier (uid) attribute for all recurring container elements. A container element is an element that contains other elements. The uid attribute is optional but is required within the context of a WITSML server. This is the equivalent of a RDBMS system generated primary key.

Low level simplistic types are defined in the following files. Each file defines many types. Each component schema file must directly or indirectly include file typ_dataTypes.xsd. If a component schema includes another component schema then it will be indirectly including typ_dataTypes.xsd. The files with "Resqml" in the name are additions to the WITSML defined types.

  • typ_dataTypes.xsd and typ_dataTypesResqml.xsd – defines simplistic data types referenced by elements and attributes in WITSML data object and component schemas. Some of these types are included from the following special data type files.
    • typ_catalog.xsd and typ_catalogResqml.xsd – defines the data types referenced by elements and attributes in RESQML data object and component schemas that are restricted content. For elements restricted by content, the valid (enumerated) values are defined in this schema.
      • typ_measureType.xsd – defines the subset of POSC measure types that are used in this specification. These types represent a numeric quantity with a unit of measure (uom) attribute. The allowed units of measure are controlled by types in file typ_quantityClass.xsd (see below).
        • typ_quantityClass.xsd – defines the enumerated POSC unit of measure acronyms for each measure type.
          • typ_baseType.xsd and typ_baseTypeResqml.xsd – defines the lowest level types. These are the types from which all other types are derived. They allow the underlying behavior of types to be controlled in one place. For example, all types are constrained to not allow an empty value and strings are constrained to not allow a blank value.

The typ_catalog.xsd and typ_catalogResqml.xsd files contain enumerations that are considered to be relatively static and should not require frequent changes. However for some data the list of recognized values may change relatively frequently. For this type of data the values are enumerated in file enumValuesResqml.xml.

3.2    WITSML Data Object Schema Design:

Since all XML documents can only have one root level element and there are situations where it is necessary to send multiple occurrences of a data object in a single XML document (e.g.; multiple wellTests), a pluralized version of the data object name was established as the schema root element. That is, the object name followed by an "S" (e.g.; productVolumes). In some cases, this leads to incorrect pluralization, but is required to order to meet the requirements of the WITSML API which generates root tag names based on the data object name. One component schema is included in all plural data object schemas:

  • cs_documentInfo.xsd - Defines information that is relevant to a specific document such as when the document was created and who was responsible for creating the document.

The unique identifiers (uid) for data objects are always defined as attributes at the singular data object name level (i.e., not the plural).

3.3    WITSML Naming Convention in Schemas

A WITSML type can be one of four kinds; a definition only with possibly a restricted size specified, a definition with enumerated values that are permitted, a reference to a component schema and a reference to a element group schema. While not really a type, the element group schemas allow the same elements to be used in multiple different contexts. In order, to help identify which type is being referenced, the following naming convention is used.

  • Type definition only - starts with lowercase, each new word begins with uppercase
    <xsd:element name="mdKickoff" type="measuredDepthCoord" minOccurs="0"/>
  • Type definition with enumerated values – starts with uppercase and each new word begins with uppercase
    <xsd:element name="shape" type="WellboreShape" minOccurs="0"/>
  • Reference to a component schema can be identified by the a "cs_" prefix.
    <xsd:element name="customData" type="cs_customData" minOccurs="0" maxOccurs="1"/>
  • Reference to an element group schema can be identified by the a "grp_" prefix.
    <xsd:group ref="grp_well" minOccurs="0" maxOccurs="1'/>
